# 下发患者完全独立复制机制说明

## 🎯 核心设计理念

**教学系统场景**：管理员创建标准病例模板，下发给所有学生学习。每个用户获得**完全独立**的副本，可以独立修改、删除，互不影响。

### 关键原则

1. ✅ **完全独立** - 副本和原始患者没有任何依赖关系
2. ✅ **独立KEY** - 每个副本有唯一的13位KEY
3. ✅ **独立删除** - 可以随时删除自己的副本，不影响他人
4. ✅ **无追溯关系** - 不记录`sourcePatientId`/`sourceRecordId`

---

## 📋 数据模型对比

### ❌ 错误的设计（有依赖关系）

```
管理员创建:
  患者A (ID=14, uniqueKey=ABC123, doctorId=4, sourcePatientId=null)
  
下发给学生B:
  患者A副本 (ID=100, uniqueKey=XYZ789, doctorId=11, sourcePatientId=14) ← 有依赖
  
问题:
  - 删除ID=14会影响ID=100
  - 需要检查副本依赖
  - 数据耦合，不够灵活
```

### ✅ 正确的设计（完全独立）

```
管理员创建:
  患者A (ID=14, uniqueKey=ABC123, doctorId=4, sourcePatientId=null)
  
下发给学生B:
  全新患者B (ID=100, uniqueKey=XYZ789, doctorId=11, sourcePatientId=null) ← 完全独立
  
下发给教师C:
  全新患者C (ID=101, uniqueKey=DEF456, doctorId=13, sourcePatientId=null) ← 完全独立

优势:
  - 三个患者完全独立
  - 可以独立删除，互不影响
  - 无需检查依赖关系
  - 数据解耦，灵活高效
```

### 3. 删除患者逻辑

**函数**: `deleteInpatientPatient`

```typescript
export const deleteInpatientPatient = async (req: AuthRequest, res: Response) => {
  const currentUser = req.user!
  const { id } = req.params
  
  const patient = await InpatientPatient.findByPk(id)
  
  // 权限检查：只能删除自己创建的患者
  if (patient.doctorId !== currentUser.id) {
    return error(res, 403, '无权操作：只能删除自己创建的患者')
  }
  
  // ✅ 注意：下发后的患者是完全独立的副本，没有依赖关系
  // 每个用户可以独立删除自己的患者，不影响其他人
  
  // 删除相关的病案
  await InpatientRecord.destroy({ where: { patientId: patient.id } })
  
  // 删除分配记录（如果有）
  await PatientAssignment.destroy({ where: { patientId: patient.id } })
  
  // 删除患者
  await patient.destroy()
  
  console.log(`✅ 用户 ${currentUser.username} 删除了患者 ID=${patient.id}`)
}
```

**关键点**：
- ✅ 只检查权限（doctorId匹配）
- ✅ 不检查副本依赖
- ✅ 直接删除，互不影响

## ⚠️ 注意事项

### 1. patient_assignments表的作用

虽然患者和病案完全独立，但`patient_assignments`表仍然有用：

**用途**：
- 记录下发历史（谁在什么时候下发了哪些患者）
- 标记是否为模板（`isTemplate`字段）
- 统计使用情况

**注意**：
- 删除患者时也会清理对应的分配记录
- 但这只是为了保持数据整洁，不影响功能

### 2. 住院号的处理

**当前策略**：
- 下发时生成新的住院号（避免重复）
- 但住院号已不再唯一（允许重复）

**未来优化**：
可以考虑让所有副本使用相同的住院号（作为模板标识）：
```typescript
inpatientNo: originalPatient.inpatientNo, // 使用原住院号
```

这样更符合"同一病例模板"的概念。

### 3. 数据一致性

**确保**：
- ✅ 每次下发都生成新的uniqueKey
- ✅ 每个副本有独立的doctorId
- ✅ 病案也完全复制并独立
