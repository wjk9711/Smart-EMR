## 🔧 调试技巧

### 如果仍然看到所有报告

#### 检查点1：确认userId是否正确

查看后端日志中的这一行：
```
用户ID: XXX
```

- 如果显示 `用户ID: undefined`，说明JWT token解析失败
- 如果显示 `用户ID: null`，说明req.user未正确设置

#### 检查点2：确认查询条件

查看后端日志：
```
查询条件: userId = XXX
```

- 确认这个ID是当前登录用户的ID
- 如果不正确，检查认证中间件

#### 检查点3：确认返回的记录

查看后端日志：
```
✅ 查询成功，返回 X 条记录
  1. ID:Y 名称:... 用户ID:Z
```

- 每条记录的 `用户ID:Z` 应该都等于查询条件的 `userId`
- 如果有不同的用户ID，说明数据库中有脏数据

#### 检查点4：直接查询数据库

```sql
-- 查看某个用户的所有报告
SELECT * FROM quality_reports WHERE user_id = 1;

-- 查看所有报告的用户分布
SELECT user_id, COUNT(*) as count 
FROM quality_reports 
GROUP BY user_id;
```

## ⚠️ 常见问题

### 问题1：所有用户都看到相同的报告

**可能原因**：
- 数据库中所有报告的 `user_id` 都是同一个值
- 查询时没有正确使用 `where: { userId }`

**解决方案**：
1. 检查数据库中报告的 `user_id` 字段
2. 确认后端代码中有 `where: { userId }`
3. 查看后端日志确认查询条件

### 问题2：用户ID为undefined或null

**可能原因**：
- JWT token无效或过期
- 认证中间件未正确设置 `req.user`
- 前端未正确传递token

**解决方案**：
1. 重新登录获取新token
2. 检查LocalStorage中的token是否存在
3. 检查Network请求中是否包含Authorization头

### 问题3：创建报告时userId错误

**可能原因**：
- 创建时使用的userId不正确
- 数据库中插入了错误的userId

**解决方案**：
1. 查看创建时的后端日志
2. 确认 `报告用户ID` 与 `当前用户ID` 一致
3. 如果不一致，检查 `QualityReport.create()` 的参数

## 🎯 总结

### 权限控制要点

1. **认证**：JWT token验证用户身份
2. **授权**：通过userId过滤数据
3. **隔离**：每个用户只能访问自己的数据
4. **审计**：通过日志追踪所有操作

### 技术实现

- **后端过滤**：`where: { userId }`
- **前端透明**：无需特殊处理，由后端保证
- **数据安全**：即使前端被篡改，后端也会正确过滤

### 最佳实践

1. ✅ 所有查询都带上userId过滤
2. ✅ 创建时自动关联当前用户
3. ✅ 删除/更新时验证所有权
4. ✅ 添加详细日志便于调试
5. ✅ 定期审计数据权限
