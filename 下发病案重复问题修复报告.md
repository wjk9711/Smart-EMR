# 下发病案重复问题修复报告

## 🐛 问题描述

**用户反馈**：下发病案时，其他用户收到的患者中，病案会重复获得一份，变成两份。

**现象**：
- 管理员下发一个包含病案的患者给用户A
- 用户A看到的患者有2份相同的病案
- 预期应该只有1份病案

---

## 🔍 问题分析

### 根本原因

**PatientAssignment表存储了两个ID**：
- `patientId`: 原始患者ID（管理员创建的患者）
- `copiedPatientId`: 副本患者ID（下发后创建的独立副本）

**问题代码**：在查询被分配患者的病案时，错误地使用了 `patientId`（原始ID），而不是 `copiedPatientId`（副本ID）。

---

### 数据流程

#### 正确的下发流程

```
1. 管理员创建原始患者 (ID=50)
   └─ 创建病案 (patientId=50)

2. 管理员下发给用户A
   ├─ 创建副本患者 (ID=100, doctorId=userA)
   ├─ 复制病案 (patientId=100) ← 关键：病案的patientId指向副本
   └─ 创建分配记录:
      - patientId: 50 (原始)
      - copiedPatientId: 100 (副本)
      - userId: userA
```

#### 错误的查询逻辑

```typescript
// ❌ 错误：使用patientId（原始ID=50）
const assignments = await PatientAssignment.findAll({
  where: { userId: currentUser.id },
  attributes: ['patientId'], // 返回 [50]
})
const assignedPatientIds = assignments.map(a => a.patientId) // [50]

// 查询病案时使用 patientId IN (50)
whereCondition.patientId = { [Op.in]: [50, ...ownPatientIds] }

// 结果：
// - 查到原始患者的病案 (patientId=50) ← 不应该看到
// - 如果用户自己也创建了病案，也会显示
// → 导致病案重复！
```

#### 正确的查询逻辑

```typescript
// ✅ 正确：使用copiedPatientId（副本ID=100）
const assignments = await PatientAssignment.findAll({
  where: { userId: currentUser.id },
  attributes: ['copiedPatientId'], // 返回 [100]
})
const assignedPatientIds = assignments.map(a => a.copiedPatientId) // [100]

// 查询病案时使用 patientId IN (100)
whereCondition.patientId = { [Op.in]: [100, ...ownPatientIds] }

// 结果：
// - 只查到副本患者的病案 (patientId=100) ← 正确
// - 加上用户自己创建的病案
// → 不会重复！
```

## ✅ 修复方案

### 修复位置1：病案列表查询

**文件**: [inpatientController.ts](file://f:/File/Project/EMR/emr-backend/src/controllers/inpatientController.ts#L298-L304)

**修改前**：
```typescript
const assignments = await PatientAssignment.findAll({
  where: { userId: currentUser.id },
  attributes: ['patientId'], // ❌ 使用原始ID
})
const assignedPatientIds = assignments.map(a => a.patientId)
```

**修改后**：
```typescript
const assignments = await PatientAssignment.findAll({
  where: { userId: currentUser.id },
  attributes: ['copiedPatientId'], // ✅ 使用副本ID
})
const assignedPatientIds = assignments
  .map(a => a.copiedPatientId)
  .filter((id): id is number => id !== null && id !== undefined) // ✅ 过滤null值
```

### 修复位置2：患者列表模板标记查询

**文件**: [inpatientController.ts](file://f:/File/Project/EMR/emr-backend/src/controllers/inpatientController.ts#L87-L100)

**修改前**：
```typescript
const templateAssignments = await PatientAssignment.findAll({
  where: {
    userId: currentUser.id,
    patientId: patientIds, // ❌ 使用原始ID
  },
  attributes: ['patientId', 'isTemplate'],
})

templateAssignments.forEach(assignment => {
  templateMap.set(assignment.patientId, assignment.isTemplate)
})
```

**修改后**：
```typescript
const templateAssignments = await PatientAssignment.findAll({
  where: {
    userId: currentUser.id,
    copiedPatientId: { [Op.in]: copiedPatientIds }, // ✅ 使用副本ID
  },
  attributes: ['copiedPatientId', 'isTemplate'],
})

templateAssignments.forEach(assignment => {
  templateMap.set(assignment.copiedPatientId!, assignment.isTemplate)
})
```

**额外修改**：将 `copiedPatientIds` 变量提升到外部作用域，以便在多个地方使用。

## 🎯 核心原则

### 数据隔离机制

```
原始患者 (doctorId=admin)
├─ ID: 50
├─ 病案: patientId=50
└─ 不能被学生/教师直接访问

副本患者 (doctorId=student)
├─ ID: 100
├─ 病案: patientId=100
└─ 学生/教师只能访问这个

PatientAssignment记录
├─ patientId: 50 (原始，用于追踪来源)
├─ copiedPatientId: 100 (副本，用于实际查询)
└─ userId: student
```

### 查询规则

1. **查询患者列表**：使用 `copiedPatientId` 获取副本患者
2. **查询病案列表**：使用副本患者的ID作为 `patientId`
3. **查询模板状态**：使用 `copiedPatientId` 匹配
4. **删除操作**：使用 `copiedPatientId` 清理分配记录

---

## ⚠️ 注意事项

### 1. 历史数据处理

如果数据库中已有旧的PatientAssignment记录，可能需要：
- 检查是否有 `copiedPatientId` 为NULL的记录
- 如果有，需要手动修复或重新下发

### 2. 前端兼容性

确保前端代码也使用正确的ID：
- 患者列表显示的应该是副本患者
- 病案编辑时使用的应该是副本患者ID

### 3. 权限验证

所有涉及患者和病案的操作都应该验证：
- `doctorId` 是否匹配当前用户
- 或者是否在 `PatientAssignment` 中有对应的 `copiedPatientId`

---

## 📝 总结

### 问题根源

**混淆了原始患者ID和副本患者ID的使用场景**：
- `patientId`: 用于追踪来源，不应该用于查询
- `copiedPatientId`: 用于实际的数据访问和权限控制

### 修复要点

1. ✅ **病案查询**：使用 `copiedPatientId`
2. ✅ **模板标记**：使用 `copiedPatientId`
3. ✅ **删除清理**：使用 `copiedPatientId`
4. ✅ **类型安全**：添加TypeScript类型守卫

### 影响范围

- **后端**：3处代码修改
- **前端**：无需修改（前端已经正确使用副本患者ID）
- **数据库**：无需修改（表结构正确）

### 最终效果

- ✅ 病案不再重复显示
- ✅ 数据隔离更加严格
- ✅ 权限控制更加准确
- ✅ 删除操作更加完整
