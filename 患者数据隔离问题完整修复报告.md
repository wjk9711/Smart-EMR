# 患者数据隔离问题完整修复报告

## ❌ 问题描述

用户反馈了两个关键问题：

1. **病案修改会同步到其他用户** - 在一个用户下修改患者信息，其他用户也能看到修改
2. **被下发的用户无法删除患者** - 提示"拒绝访问 Request failed with status code 403"

## 🔍 根本原因分析

### 问题1：查询逻辑错误导致数据不独立

**核心问题**：下发患者时创建了完全独立的副本，但查询时使用的是原始患者ID，导致所有用户看到的都是同一个患者。

#### 错误的查询逻辑（修复前）

```typescript
// inpatientController.ts - getInpatientPatients函数

// 学生和教师：查看自己创建的 + 被分配的
const assignments = await PatientAssignment.findAll({
  where: { userId: currentUser.id },
  attributes: ['patientId'], // ❌ 获取的是原始患者ID
})
patientIds = assignments.map(a => a.patientId)

if (patientIds.length > 0) {
  whereCondition[Op.or] = [
    { doctorId: currentUser.id },
    { id: { [Op.in]: patientIds } } // ❌ 查询原始患者，不是副本！
  ]
}
```

**问题分析**：
- 查询时使用`patientId=18`（原始患者）
- 但应该使用`copied_patient_id=20`（副本患者）
- 结果：所有用户都看到了ID=18的原始患者
- 修改ID=18会影响所有人

### 问题2：部分下发失败导致无效记录

**现象**：只有chenmj（userId=12）有副本患者，student1和teacher001没有副本。

**原因**：之前的下发操作可能因为某些原因（事务回滚、字段不存在等）只成功了一部分。

**后果**：
- `PatientAssignment`表中有记录，但`copied_patient_id`为null
- 这些用户实际上没有收到副本患者
- 但仍然尝试查询原始患者，导致权限混乱

## 🛠️ 修复方案

### 修复1：更正查询逻辑

**文件**: `emr-backend/src/controllers/inpatientController.ts`

**修改内容**:

```typescript
// 修复后：使用copiedPatientId而不是patientId
const assignments = await PatientAssignment.findAll({
  where: { userId: currentUser.id },
  attributes: ['copiedPatientId'], // ✅ 获取副本患者ID
})

// 过滤掉null值
const copiedPatientIds = assignments
  .map(a => a.copiedPatientId)
  .filter(id => id !== null) as number[]

if (copiedPatientIds.length > 0) {
  whereCondition[Op.or] = [
    { doctorId: currentUser.id },
    { id: { [Op.in]: copiedPatientIds } } // ✅ 查询副本患者
  ]
} else {
  whereCondition.doctorId = currentUser.id
}
```

**效果**：
- ✅ 每个用户只能看到自己的副本患者
- ✅ 修改副本不会影响其他用户
- ✅ 数据完全隔离

### 修复2：清理无效的分配记录

**效果**：
