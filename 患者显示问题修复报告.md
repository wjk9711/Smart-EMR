# 患者显示问题修复报告

## 问题描述

1. **teacher001用户下的"小丽"患者不显示**
2. **其他用户也看不到自己创建的患者**
3. **刷新时提示Request failed with status code 500**

## 问题原因分析

### 根本原因：Sequelize查询操作符使用错误

在实现数据隔离功能时，使用了错误的Sequelize操作符：

#### 错误代码1：患者列表查询
```typescript
// ❌ 错误：使用了Symbol.for('or')，Sequelize不支持
whereCondition[Symbol.for('or')] = [
  { doctorId: currentUser.id },
  { id: patientIds }  // ❌ 错误：应该使用Op.in
]
```

#### 错误代码2：病案列表查询
```typescript
// ❌ 错误：直接赋值数组，虽然Sequelize能识别，但不规范
whereCondition.patientId = accessiblePatientIds
```

### 导致的症状

1. **500错误**：`Symbol.for('or')`不是有效的Sequelize操作符，导致SQL语法错误
2. **患者不显示**：查询条件无法正确构建，导致WHERE子句错误
3. **所有用户受影响**：学生和教师的查询逻辑都使用了错误的操作符

## 修复方案

### 1. 导入Sequelize Op操作符

```typescript
import { Op } from 'sequelize'
```

### 2. 修复患者列表查询逻辑

**修改前：**
```typescript
whereCondition[Symbol.for('or')] = [
  { doctorId: currentUser.id },
  { id: patientIds }
]
```

**修改后：**
```typescript
whereCondition[Op.or] = [
  { doctorId: currentUser.id },
  { id: { [Op.in]: patientIds } }
]
```

### 3. 修复病案列表查询逻辑

**修改前：**
```typescript
whereCondition.patientId = accessiblePatientIds
```

**修改后：**
```typescript
whereCondition.patientId = { [Op.in]: accessiblePatientIds }
```

## 关于唯一键值的说明

### 当前数据库设计

#### 1. 患者表的唯一约束

**inpatient_patients表：**
- `id`: 主键，自增，全局唯一
- `inpatientNo`: 住院号，**UNIQUE约束**，全局唯一
- `idCard`: 身份证号，**UNIQUE约束**，全局唯一

**patients表（门诊）：**
- `id`: 主键，自增，全局唯一
- `patientNo`: 病历号，**UNIQUE约束**，全局唯一

#### 2. 病案表的唯一约束

**inpatient_records表：**
- `id`: 主键，自增，全局唯一
- `caseNo`: 病案号（与患者住院号相同）
- **联合唯一约束**: `(patientId, recordType)` - 每个患者的每种病案类型只能有一条记录

### 管理员下发患者时的键值冲突分析

#### ✅ 不会发生冲突的原因

1. **患者ID是全局唯一的**
   - 患者通过自增ID标识，不同用户创建的患者ID不同
   - 下发时只是创建`patient_assignments`记录，不复制患者数据
   - 原始患者记录保持不变

2. **住院号是全局唯一的**
   - 每个患者有唯一的住院号（inpatientNo）
   - 下发不改变住院号
   - 多个用户看到的是同一个患者记录（通过分配关系）

3. **病案ID是全局唯一的**
   - 病案通过自增ID标识
   - 病案关联到患者（patientId），而不是直接关联到医生
   - 下发患者后，该患者的所有病案对接收者可见

4. **病案号的特殊性**
   - 病案号（caseNo）与患者住院号相同
   - 同一患者的所有病案共享相同的病案号
   - 通过`(patientId, recordType)`联合唯一约束确保不重复

#### 数据流向示例

```
管理员创建患者A (ID=5, inpatientNo='2024051001', doctorId=4)
    ↓
管理员下发患者A给用户B (teacher001, ID=13)
    ↓
创建分配记录: patient_assignments(patientId=5, userId=13, assignedBy=4)
    ↓
用户B查询患者列表:
  - WHERE doctorId=13 OR id IN (5)
  - 结果：看到患者A（因为id=5在分配列表中）
    ↓
用户B查看患者A的病案:
  - WHERE patientId IN (5)
  - 结果：看到患者A的所有病案
```

### 潜在风险点及防护

#### 风险1：住院号重复

**场景**：两个用户创建了相同住院号的患者

**防护**：
- 数据库层面：`inpatientNo`有UNIQUE约束
- 应用层面：创建患者时自动生成唯一住院号
- 结果：不可能出现重复住院号

#### 风险2：病案类型重复

**场景**：同一患者创建了两个相同类型的病案

**防护**：
- 数据库层面：`(patientId, recordType)`联合唯一约束
- 应用层面：创建病案前检查是否已存在
- 结果：每个患者的每种病案类型只能有一条

#### 风险3：分配记录重复

**场景**：管理员多次下发同一患者给同一用户

**防护**：
- 数据库层面：`(patientId, userId)`联合唯一索引
- 应用层面：`bulkCreate`时使用`ignoreDuplicates: true`
- 结果：重复下发不会报错，也不会创建重复记录
